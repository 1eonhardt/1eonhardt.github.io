版本控制

当你开发一个简单的小项目的时候，可能没几个文件，记住开发进度和上次修改了什么比较轻松。但是版本控制(tracking revisions over time,按时间追踪记录修改)会随着项目复杂化而更加复杂。

如果你不仅想追踪修改历史，还想知道是谁，什么时候修改的。就需要版本控制系统(Version Control System,VCS)。

VCS软件(以git为例)，通常包含以下工具：

  • 保存项目状态 (snapshot)
  • 查看修改历史 (git log)
  • 撤销修改 (git reset)
  • 防止冲突

git是当前最流行的版本控制工具，用a stream of snapshot记录修改，并且是分布式的。

github或coding.net是提供git仓库托管服务的。你可以将你本地的仓库推送到github,github中就有了和你本地仓库完全一样的库。在github上，有图形化界面可以方便的进行修改，合并等操作。

使用Git

初始化仓库(init repository)

首先，创建一个新文件夹，并进入该文件夹。或者进入一个已有的文件夹。
执行git init命令即可。
执行完之后，该文件夹下会出现一个.git文件夹。本地仓库初始化就成功了。

跟踪文件(tracking files)，提交(commit)，查看提交历史

库(即init的文件夹)中的文件有三种状态：

  • Not tracked:未跟踪，git不会记录这类文件的变化
  • Staged:暂存，变化已经记录到临时工作区(Staging Area)
  • Committed:已提交，记录修改确定了，git中记录的当前版本就是提交后的版本了

第一次提交时，需要设置用户名和邮箱地址。多人协作的必要步骤。

分支(branch)

Stash

git stash命令会暂时缓存任何你未提交的修改，包括暂存区的和未暂存的(没有通过git add命令添加到暂存区的，包括新add的增删文件，修改add过的文件)，不包括未追踪的文件。stash就相当于将之前的修改都隐藏了起来，之后的操作不会受到之前修改的限制。
注意：stash只保留在本地库中，不会被推送到远程库。

注意：git stash apply会将修改应用到所有分支。比如，在test分支修改了一个文件，stash以后再apply回来，这样master分支中也会有一个未提交的修改。

合并冲突

在合并分支，apply stash，pull，push时如果文件内容有冲突(即不同人/分支的不同提交中修改了相同的地方)。

解决冲突：首先用git status查看有冲突的文件，打开有冲突的文件，在有冲突的地方会有提示符。<<<<<<<, ========, 和>>>>>>>。一般 <<<<<<< 和 ======= 之间是接受合并的分支，另一半是被合入的分支。把冲突的地方改好后提交即可。
